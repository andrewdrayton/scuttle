# Architecture

Scuttle is built around three memory files and two search commands. The skills contain the logic; the memory files contain everything that makes it yours.

---

## The Three Memory Files

```
{scuttle.memory_path}/
├── job-search-profile.md    ← who you are, what you want, how to score
├── job-search-patterns.md   ← what the skill has learned from your feedback
└── job-search-sources.md    ← verified ATS endpoints and curated boards
```

### `job-search-profile.md`
The only file you write manually. Defines your target roles, target companies, industries, location rules, hard exclusions, and the scoring framework. Read at the start of every search session. The skill does not modify this file — it only reads it.

### `job-search-patterns.md`
Written and maintained by the skill. Starts empty. Grows over time as you give feedback after each report. Contains:
- **Liked** — roles you applied to or reacted positively to
- **Disliked** — roles you passed on with reasons
- **Active Filters** — auto-extracted patterns the skill applies on every future run ("skip CoS roles requiring consulting/IB background")
- **Warm Approaches** — companies you're pursuing through a network path
- **Dedup table** — every role ever surfaced, so the same role is never shown twice

Do not edit this file manually unless correcting an error. The skill owns it.

### `job-search-sources.md`
Written and maintained by the skill. Contains:
- **Verified ATS Endpoints** — companies with working API URLs, verified on a specific date
- **Discovered Career Pages** — career pages found but ATS type unknown, queued for future discovery
- **Failed Sources** — companies where discovery failed, with date and reason (never deleted — prevents re-attempting dead ends)
- **Curated Boards** — Tier 1 job boards seeded from your profile

You seed this file initially with companies you want monitored. The skill discovers and adds more over time.

---

## The Two Commands

### `scuttle` — targeted search
Hits your verified ATS endpoints and curated boards. Fast, bounded, reproducible.

```
Read profile + patterns + sources
         ↓
Batch-fetch all ATS endpoints (one Python script)
         ↓
Run Tavily searches on curated boards (max 3)
         ↓
Apply hard exclusions → apply learned filters → score
         ↓
Deliver structured report
         ↓
User gives feedback → skill writes to patterns.md
```

Token budget: 35–45K tokens per run. Hard cap 55K.

### `dig` — broad sweep
Casts a wider net via signal-based Tavily queries across aggregators. Designed to surface roles that would never appear in a standard ATS search — founder-proximate ops, talent-adjacent roles, pre-infrastructure hires at newly-funded companies.

Results are staged to `{scuttle.memory_path}/dig-staging.md` rather than delivered inline. Review them with the `digest` command when ready.

Token budget: 25–40K tokens. Hard cap 50K. Run at most once per week.

## The Learning Loop

This is what separates Scuttle from a static job search wrapper.

```
Session 1:  Report delivered → user gives feedback
                                        ↓
                            Skill writes to patterns.md
                                        ↓
Session 2:  Skill reads patterns.md → applies learned filters
            → boosts roles matching liked patterns
            → auto-skips roles matching disliked patterns
            → surfaces adjacent companies to liked ones
```

After 3–5 sessions the signal accumulates enough to meaningfully change what gets surfaced. The patterns file is the memory — the longer you use it, the more calibrated it gets.

---

## Source Discovery

Sources grow over time through two mechanisms:

**Explicit:** User mentions a company ("check Madhappy's jobs") → skill runs discovery workflow → detects ATS type → verifies API URL → writes to sources file.

**Implicit:** Adjacency search surfaces a company the skill hasn't seen → skill opportunistically discovers its ATS endpoint → adds to sources file.

Discovery is deliberately rate-limited: max 5 discovery operations per session, never during a `scuttle` run. Discovery is expensive — spread it across sessions.

---

## The Three-Tier Source Hierarchy

| Tier | What | Why |
|------|------|-----|
| Tier 1 | Curated industry job boards (Tavily site: queries) | High signal, industry-specific, low noise |
| Tier 2 | Verified ATS endpoints (Greenhouse, Lever, Ashby, etc.) | Reliable, programmatic, filterable |
| Tier 3 | Broad aggregators (Google Jobs, LinkedIn) | Last resort only — high noise |

Scuttle always starts with Tier 1 and 2. Tier 3 only if fewer than 5 candidate roles surface.

---

## Response Size Design

ATS endpoints can return 5MB+ of raw JSON. Scuttle routes every ATS fetch through a Python script that filters to relevant titles before bringing data into context. A naive fetch of Alo Yoga returns 652 jobs; the filtered fetch returns 3–8 relevant ones.

This keeps token costs bounded and prevents context blowouts on large career sites.

## Skill Structure

```
scuttle/SKILL.md          ← entry point, triggers on "scuttle" command
job-search/SKILL.md       ← core search logic, learning loop, source discovery
dig/SKILL.md              ← broad aggregator sweep, digest command
```

`scuttle` is the user-facing entry point. It loads `job-search` via `skill_view('job-search')` and delegates execution. This separation keeps the entry point minimal and lets `job-search` be loaded independently for ad-hoc use.
